# Cursor에서의 하네스 엔지니어링 (Karpathy 정렬)

이 문서는 **Cursor** 안에서 Andrej Karpathy가 지적한 LLM 코딩 실패 패턴을 줄이기 위해, [그의 관찰](https://x.com/karpathy/status/2015883857489522876)에 맞춘 **하네스(행동 제약)**를 어떻게 걸고 유지하는지 설명한다. 원칙 본문·예시·표는 루트의 [`CLAUDE.md`](CLAUDE.md)를 본문으로 삼는다.

## 하네스 엔지니어링이란

에이전트에게 “똑똑해져라”만 말하기보다, **반복해서 적용되는 짧은 규칙**으로 다음을 강제하는 것이다.

- 묵시적 가정·숨은 혼란 금지, 트레이드오프 드러내기  
- 최소 코드·최소 diff  
- 검증 가능한 성공 기준(테스트·재현 등)으로 일 돌리기  

Karpathy식 요지는 한 문장으로 말하면, **명령 나열**보다 **성공 기준 + 검증 루프**를 주는 쪽이 모델이 스스로 맞출 때까지 돌기 쉽다는 것이다. 이는 [`CLAUDE.md`](CLAUDE.md)의 “목표 중심 실행”과 “핵심 통찰”에 정리되어 있다.

## 이 저장소에 이미 있는 것

| 파일 | 역할 |
|------|------|
| [`CLAUDE.md`](CLAUDE.md) | 네 원칙 전체, 배경 인용, 표, 프로젝트 병합 예시. **에이전트·사람 모두**가 읽는 단일 근거 문서. |
| [`.cursorrules`](.cursorrules) | Cursor가 프로젝트를 열 때 함께 읽는 **짧은 하네스**. `CLAUDE.md`와 겹치면 요약만 유지하고 상세는 `CLAUDE.md`로 맡긴다. |

두 파일은 **같은 Karpathy 정렬**을 목표로 하며, `.cursorrules`의 “충돌 시” 절에 따라 사용자의 **더 구체적·최근** 지시가 우선한다.

## Cursor에서 규칙이 걸리는 위치 (개념)

1. **프로젝트 루트** — `.cursorrules`, (선택) `.cursor/rules/*.mdc`  
2. **사용자 전역** — Cursor 설정의 **Rules / User Rules** (모든 저장소에 공통으로 걸고 싶을 때)  

Cursor는 기본적으로 **`CLAUDE.md`를 자동으로 읽지 않는다.** Claude Code용 관례 이름이므로, **Cursor 쪽 하네스**로 쓰려면 아래 중 하나를 택하면 된다.

- **권장:** `CLAUDE.md` 내용을 **프로젝트 규칙**으로 끌어온다.  
  - 예: `.cursor/rules/karpathy-harness.mdc`를 만들고 `alwaysApply: true`로 두고, 그 안에 네 원칙 요약 또는 `CLAUDE.md`와 동일한 본문을 넣는다.  
- **대안:** 지금처럼 **`.cursorrules`**에 요약을 두고, 사람·에이전트는 필요할 때 `CLAUDE.md`를 명시적으로 참조한다.

## 다른 프로젝트에 같은 하네스 옮기기

1. 이 저장소의 [`CLAUDE.md`](CLAUDE.md)를 복사하거나, 기존 지침 파일 뒤에 **append**한다.  
2. Cursor용으로는  
   - **A)** `.cursorrules`에 [`.cursorrules`](.cursorrules)와 같은 형태의 요약을 복사하거나  
   - **B)** `.cursor/rules/` 아래 `.mdc` 규칙 파일을 만들고 `alwaysApply: true`로 켠다.  
3. 프로젝트만의 규칙(스택, 린트, API 관례)은 **별 섹션**으로 추가해 Karpathy 본문과 **병합**한다. 원칙 텍스트를 임의로 희석하지 말고, “프로젝트별 지침” 블록을 아래에 붙이는 방식이 안전하다.

## 사용자(사람)가 하네스를 쓰는 법

- 작업 요청을 **검증 가능한 목표**로 바꿔 보낸다.  
  - 예: “버그 고쳐줘” → “이 입력에서 재현되는 테스트를 추가하고 통과시켜줘.”  
- 모호하면 에이전트가 질문하도록 **의도적으로 정보를 나눠** 보내지 않는다. 한 번에 스펙을 주는 편이 diff와 재작성 비용을 줄인다.  
- “그냥 리팩터링도 해줘”처럼 범위를 넓히면 **정밀한 수정** 원칙과 충돌한다. 필요하면 별 요청으로 분리한다.

## Claude Code와의 차이

| 도구 | 흔한 관례 |
|------|-----------|
| **Claude Code** | 저장소 루트 `CLAUDE.md`, 플러그인/스킬 마켓플레이스 등 |
| **Cursor** | `.cursor/rules`, `.cursorrules`, 설정의 User Rules |

같은 **네 원칙**을 쓰더라도, **파일 이름과 배치**는 도구에 맞게 나누는 것이 하네스 엔지니어링이다.

## 유지보수 (기여 시)

네 원칙 문구를 바꿀 때는 다음을 **한 세트**로 맞춘다.

- [`CLAUDE.md`](CLAUDE.md) — 전체 근거  
- [`.cursorrules`](.cursorrules) — Cursor용 요약 하네스  

요약만 고치고 본문을 어긋나게 두지 않도록 한다.

---

**요약:** Karpathy 의도대로의 하네스는 “짧은 반복 규칙(Cursor 쪽) + 긴 근거 문서(`CLAUDE.md`)”의 조합이다. Cursor는 `CLAUDE.md`에 자동 의존하지 않으므로, 규칙 UI나 `.cursor/rules`로 **한 번 더 걸어주는 것**이 의도에 가깝다.
